O UniMaps resolve um problema que o GPS não consegue: orientar pessoas dentro de um prédio. Este material explica, em profundidade, como ele foi construído — do grafo e do algoritmo de rotas à infraestrutura que roda no próprio navegador.

role para explorar o sistema

O ponto de partida

Por que o GPS não basta

Dentro de um prédio, o sinal de satélite é bloqueado por lajes e paredes. Resultado: a navegação que funciona na rua simplesmente não funciona no corredor — exatamente onde estudantes, visitantes e candidatos se perdem.

desafio

GPS falha indoor

Posicionamento por satélite não atravessa a estrutura do edifício. Não há sinal confiável dentro das salas.

restrição

Sem infraestrutura cara

Beacons e sensores exigem instalar e manter dezenas de dispositivos. Inviável para uma instituição comum.

objetivo

Acessível a todos

Tem que funcionar em qualquer celular, sem instalar nada, e incluir pessoas com deficiência.

por que A*

Rápido e ótimo

O A* combina o custo já percorrido com uma estimativa (heurística) do que falta. Com uma heurística admissível — aqui, a distância em linha reta — ele garante o menor caminho explorando bem menos nós que uma busca cega.

o truque da acessibilidade

Mudar o grafo, não o algoritmo

A rota acessível não é um algoritmo separado: o sistema remove as arestas de escada do grafo e roda o mesmo A*. Elegante e fácil de manter — a regra de negócio vira topologia.

Arquitetura

Quase tudo roda no seu celular

O servidor só entrega arquivos e guarda dados — não processa rota. Toque em cada camada para abrir os detalhes técnicos.

No navegador (cliente)onde o trabalho acontece ▾
Interface 3DHTML/CSS/JS puro, sem framework, em arquivo único. 3D com Three.js r160 via CDN (importmap).
Cérebro: A*Grafo + A* em JavaScript. Cada celular calcula a própria rota — nada vai a servidor.
Service Worker (PWA)Cacheia o app na 1ª visita. Páginas usam network-first (sempre a versão nova online); offline, usa o cache.
Voz e controleInstruções por Web Speech API. Suporte a controle de Xbox via Gamepad API na câmera livre.
Na nuvem (entrega e dados)infraestrutura ▾
Vercel · hospedagemSite estático servido por CDN global. Deploy contínuo: git push publica a versão nova sozinho.
Supabase · dadosPostgreSQL com 3 tabelas — blocos, nos, arestas — e API REST automática. Leitura pública via RLS.
Rede de segurançaSe o Supabase cair, o app usa a cópia local embutida e funciona igual. Sem ponto único de falha no uso.
SegurançaCabeçalhos no vercel.json: Content-Security-Policy (CSP) e HSTS restringem origens e forçam HTTPS.
Versionamento e entregaprocesso ▾
GitHubCódigo versionado com commits semânticos. Cada build carrega um marcador de versão no rodapé para conferência.
CI/CDIntegração GitHub → Vercel: o push dispara o build e a publicação automaticamente, sem etapa manual.

Inclusão por projeto

Acessibilidade não é enfeite

Recursos pensados para diferentes necessidades — e todos funcionam offline.

🦽 Rota acessível

Remove a escada do grafo e roteia pela rampa, para cadeirantes e mobilidade reduzida.

🔊 Leitura por voz

As instruções da rota são faladas em português (Web Speech API), para baixa visão.

📝 Legendas grandes

Instruções destacadas em texto grande na tela, para pessoas surdas — sem depender de internet.

🎨 Alto contraste

Modo de contraste reforçado e três tamanhos de fonte para a interface inteira.

🎬 Reduzir animações

Desliga movimentos para quem tem sensibilidade vestibular.

⚡ Velocidade do trajeto

Ajuste de 0,5× a 2× no ritmo do guia, para acompanhar com conforto.

Da feira ao mercado

Maturidade do produto

O sistema foi projetado para escalar de um pavimento a um produto comercial. As fases 1 a 4 são roadmap.

Fase 0 · hoje
Protótipo da feira
App 3D, A*, QR Code, PWA offline e acessibilidade — um pavimento, pronto para demonstração.
Fase 1
Editor visual de mapas
Desenhar salas e corredores sobre a planta gera o grafo. Mapear vira tarefa de mouse, não de código.
Fase 2
Campus completo
Múltiplos andares e prédios na mesma engine; escadas e elevadores como conexões entre pavimentos.
Fase 3
Automação do mapeamento
Conversor de planta DWG e OCR dos nomes de sala reduzem o mapeamento de dias para horas.
Fase 4
Produto comercial (SaaS)
Back-end multi-cliente, login e versionamento de mapas — sob arquitetura hexagonal — para hospitais, universidades e shoppings.

Tecnologias da fase 2, por frente

mapeamento

Editor visual · conversor DWG · OCR de plantas · navmesh.

localização

Dead reckoning por sensores do celular · beacons BLE (premium).

produto

Multi-tenant · login (Auth) · versionamento · arquitetura hexagonal.

qualidade

Assistente de IA · analytics de uso · testes automatizados.

O que o UniMaps NÃO usa — porque saber os limites é parte da engenharia:
  • Beacon / GPS: a localização é por QR Code. Ele sabe onde você começou, não rastreia em tempo real.
  • Java / Spring Boot: não há back-end próprio. A lógica é JavaScript no navegador.
  • AWS: a hospedagem é Vercel + Supabase.
  • Arquitetura hexagonal: o app atual é uma SPA cliente; o padrão hexagonal entra no back-end comercial da fase 4. (Hoje há, sim, um padrão Adapter na camada de dados: Supabase ou cópia local atrás da mesma interface.)
